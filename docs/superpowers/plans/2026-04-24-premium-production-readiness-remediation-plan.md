# ComplianceOS Premium Production Readiness Remediation Plan

## Project Summary
ComplianceOS target is an India-first compliance platform with an event-sourced double-entry accounting core, strict tenant isolation, GST and ITR processing, payroll, inventory, invoicing, and onboarding flows that are reliable under real production conditions. Success means the system preserves accounting integrity, enforces tenant-safe access controls, keeps projector-driven read models consistent with immutable events, and provides deployable operations guardrails including CI gates, monitoring, backup and restore confidence, and rollback safety. The current codebase delivers broad functional coverage, but premium-grade production readiness is not yet achieved because key security boundaries, event processing contracts, and automated quality controls remain incomplete or inconsistent.

## Code Review Findings
Issue CR-01 at `apps/web/middleware.ts` is Critical because path protection uses `pathname.startsWith("/(app)")`, which does not match real Next.js route-group URLs and therefore bypasses intended auth and onboarding enforcement; recommended fix is explicit protected-path matching and server-side auth checks in layout, estimated effort 0.5 day. Issue CR-02 at `apps/web/app/(app)/layout.tsx` is Critical because the layout is client-only and does not enforce server auth even though `requireAuth` exists in `apps/web/lib/session.ts`; recommended fix is adding server auth redirect in the app shell before render, estimated effort 0.5 day. Issue CR-03 at `apps/web/lib/auth.ts` is High because demo login accepts `demo@complianceos.test` without password verification; recommended fix is environment-gated demo mode and proper hashed password verification for all accounts, estimated effort 0.5 to 1 day. Issue CR-04 at `packages/server/src/routers/accounts.ts`, `packages/server/src/routers/journal-entries.ts`, `packages/server/src/routers/balances.ts`, `packages/server/src/routers/fiscal-years.ts`, and `packages/server/src/routers/gst-returns.ts` is High because sensitive routes still use `publicProcedure` while mutating code uses `ctx.session!`; recommended fix is converting to `protectedProcedure` and applying role checks for mutations, estimated effort 1 to 2 days. Issue CR-05 at `packages/server/src/routers/itr-payment.ts` and `packages/server/src/routers/itr-computation.ts` is High because caller-supplied `tenantId` can override context and open IDOR risk; recommended fix is deriving tenant only from authenticated context for user-facing procedures, estimated effort 0.5 day. Issue CR-06 at `packages/server/src/routers/onboarding.ts` is High because onboarding operations trust input `tenantId` without explicit ownership or membership checks; recommended fix is mandatory `user_tenants` membership and role validation before every tenant read or write, estimated effort 1 day. Issue CR-07 at `packages/db/migrations/0001_add_rls_policies.sql` is High because RLS is enabled but not hardened with `FORCE ROW LEVEL SECURITY` and policy coverage for tenant updates is incomplete versus onboarding usage; recommended fix is hardened RLS migration and tenant table policy verification, estimated effort 1 to 2 days. Issue CR-08 at `packages/server/src/lib/event-store.ts` and `packages/server/src/projectors/worker.ts` is Critical because sequence is written per aggregate but consumed as global cursor, causing possible event skips and replay defects; recommended fix is selecting one sequence model and migrating cursor logic accordingly, estimated effort 3 to 5 days. Issue CR-09 at `packages/server/src/projectors/worker.ts` is Critical because projector event fetch is not tenant-scoped while cursor state is tenant-scoped; recommended fix is tenant-filtered fetch and cursor updates scoped to tenant batches, estimated effort 1 day. Issue CR-10 at `packages/server/src/projectors/worker.ts` is Critical because the worker does not implement `FOR UPDATE SKIP LOCKED` queue semantics promised by spec; recommended fix is locked batch acquisition with safe multi-worker behavior, estimated effort 3 to 4 days. Issue CR-11 at `packages/server/src/projectors/worker.ts` is Critical because event-processing errors are logged but cursor still advances, creating silent permanent event loss; recommended fix is fail-fast cursor semantics with dead-letter and retries, estimated effort 1 to 2 days. Issue CR-12 at `packages/server/src/projectors/worker.ts` is High because worker startup depends on existing `projector_state` rows with no robust bootstrap path for new tenants; recommended fix is deterministic projector state initialization, estimated effort 0.5 to 1 day. Issue CR-13 at `packages/server/src/commands/post-invoice.ts` and projector consumers like `packages/server/src/projectors/receivables-summary.ts` and GST projectors is Critical because event payload contracts are inconsistent; recommended fix is shared typed payload schemas with compile-time contract checks and projector contract tests, estimated effort 3 to 5 days. Issue CR-14 at GST projectors and schema (`packages/server/src/projectors/gst-itc-available.ts`, `packages/server/src/projectors/gst-liability.ts`, `packages/db/src/schema/gst-ledgers.ts`) is High because upsert target columns do not have aligned unique constraints; recommended fix is adding unique constraints matching upsert targets and reconciling existing rows, estimated effort 1 to 2 days. Issue CR-15 at entry numbering (`packages/server/src/lib/entry-number.ts` and commands invoking it) is High because counter increments may occur outside enclosing business transaction and can create numbering gaps on failure; recommended fix is transaction-bound counter locking and increment, estimated effort 1 to 2 days. Issue CR-16 at journal commands (`post/void/modify/delete`) is High because race windows exist from read-then-update without row-level lock or conditional status update; recommended fix is atomic update with status predicates and affected-row validation, estimated effort 1 to 2 days. Issue CR-17 across frontend and API wiring is Critical because `/api/trpc` route implementation is not clearly present while many pages directly call it; recommended fix is explicit App Router tRPC handler and integration tests validating one read and one write path, estimated effort 1 to 2 days. Issue CR-18 at `apps/web/app/(auth)/signup/page.tsx` is High because it posts to `/api/auth/register` but no route exists; recommended fix is implementing register endpoint or removing unsupported flow and aligning UI to actual auth mechanism, estimated effort 0.5 day. Issue CR-19 across web and server is High because type-safety suppression is widespread (`@ts-nocheck`, `ts-ignore`) and Next build currently ignores type errors in `apps/web/next.config.ts`; recommended fix is phased strictness restoration with gates and suppression burn-down, estimated effort 2 to 3 weeks phased. Issue CR-20 in repo-level automation is High because CI workflows are missing, root test scripts are missing, turbo lacks `test` orchestration, and server package has tests without reliable package-level test execution contract; recommended fix is full CI and test pipeline normalization, estimated effort 2 to 3 days. Issue CR-21 in repository hygiene is Medium because a large volume of AppleDouble or junk artifacts (`._*`, `.!*`) pollute source and test discovery; recommended fix is one-time cleanup, ignore hardening, and pre-commit checks, estimated effort 1 to 2 days. Issue CR-22 in operations is High because observability is minimal and health checks are shallow (`apps/web/app/api/health/route.ts`), with no alerting SLOs or projector lag telemetry; recommended fix is structured logs, metrics, dependency-aware health checks, and alert runbooks, estimated effort 4 to 6 days.

## Testing Strategy
Testing must move to a layered, repeatable model. Unit tests should cover deterministic business logic and invariants including debit-credit checks, fiscal year closure rules, numbering behavior, tax calculators, event payload schema validation, and status transitions; these should use Vitest with deterministic fixtures and explicit boundary values including zero, maximum precision, invalid enums, and closed-state mutation attempts. Integration tests should run on ephemeral PostgreSQL and Redis test environments with migrations applied per run and seeded deterministic tenants, then validate full command and projector effects in one flow, including event append, projection updates, and cursor movement with rollback assertions. System tests should run end-to-end flows via Playwright covering onboarding, journal entry lifecycle, invoice post and payment lifecycle, payroll process and finalize, GST return generation, ITR computation and payment, and all associated failure paths such as unauthenticated access, wrong tenant identifiers, stale transitions, and duplicate submit behavior. Concurrency tests should stress race windows with parallel posting and voiding, parallel counter increments, and worker concurrency to validate no duplicate numbers, no missed events, and no double-application of projections. Performance tests should use k6 with explicit targets such as p95 API latency, projector lag under burst load, and report query response ceilings under realistic tenant and transaction volumes, with baseline snapshots and regression thresholds. Required test data should include clean base fixtures, compliance-specific tax cases, large-volume invoice and payroll datasets, and intentionally malformed event payloads for resilience testing. Required mocks should include SMTP, storage provider, OCR service, and time control to stabilize fiscal and due-date behavior. Automation framework should standardize on Vitest plus Testcontainers for backend/integration, Playwright for browser/system, and k6 for load, all orchestrated in CI.

## Gap Analysis
Feature and architecture gaps exist where spec commitments are not met in implementation, especially in event queue semantics, tenant-safe projector processing, and strict auth boundaries. Documentation gaps include missing `ARCHITECTURE.md` despite README references, partially aspirational deployment guidance, and operational instructions that do not match current automated reality. Configuration gaps include absent GitHub workflows, absent root-level test commands despite docs claiming them, turbo pipeline without `test`, and production build configuration allowing suppressed type errors. Operational gaps include no robust logging schema, limited health checks without dependency validation, no alert policy definitions, no projector lag metrics, no dead-letter queue strategy, and no incident response runbook tied to measurable SLOs. Backup and disaster recovery gaps include no documented tested restore drill cadence, no explicit RPO or RTO values, and no deploy migration rollback playbook beyond general commands. Security and compliance gaps include inconsistent route authorization, demo auth behavior that must be environment-gated, incomplete tenant ownership checks in onboarding and ITR paths, and RLS hardening tasks still pending. Maintainability gaps include heavy `@ts-nocheck` debt, inconsistent coding conventions across modules, and source tree contamination by resource-fork artifacts that increase review noise and CI fragility.

## Production Readiness Checklist
Environment parity must be verified across local, staging, and production for Node version, schema migration state, auth mode, queue semantics, and feature-flag behavior. CI and CD must include required status checks for lint, typecheck, unit tests, integration tests, system smoke tests, and build; deployments must use immutable version tags and block promotion on failed checks. Versioning discipline must include release notes and traceability from commit to artifact to deployment. Rollback procedure must be explicit and tested: application rollback by image or tag, database safety by pre-deploy snapshot plus forward-fix migration strategy and validated restore commands. Security hardening must include protected procedures for all tenant-scoped routes, elimination of input-driven tenant overrides, secure demo-account controls, strict secret handling, dependency vulnerability scans, and least-privilege database roles with enforced RLS. Compliance checks must include accounting invariants, immutable event auditability, GST and ITR regression fixtures, fiscal-year business-rule verification, and retention and export behavior. Performance benchmarks must be defined, measured, and tracked over time with accepted p95 targets and projector lag thresholds. Observability must include structured logs, trace correlation IDs, dependency-aware health checks, metrics dashboards, and alert rules with documented escalation and on-call actions. Disaster readiness must include scheduled backup restore drills, clear RPO and RTO definitions, and a documented outage communication path.

## Execution Guide for Junior Developer
Begin by creating a new branch dedicated to hardening and create a work log file that records each completed checkpoint and command output location. Install prerequisites in this order: Node 20+, pnpm 9+, Docker, PostgreSQL 16, Redis 7, Vitest, Playwright, and k6, then verify versions before touching code. First, fix security boundaries: repair middleware path protection, enforce server auth in app layout, convert unsafe public procedures to protected ones, and remove user-input tenant overrides; verify by running manual unauthorized route checks and writing integration tests proving blocked access. Second, repair projector and event correctness: align event sequencing model, scope worker reads by tenant, implement `FOR UPDATE SKIP LOCKED`, stop cursor advancement on processing error, and initialize projector state for new tenants; verify by replay tests that intentionally inject failures and confirm no event loss. Third, harden concurrency and transaction integrity: make numbering updates part of the same business transaction, add atomic status transition updates with predicates, and run race-condition tests for journal and invoice lifecycles; verify no duplicates and no invalid state transitions after parallel runs. Fourth, fix API runtime completeness: ensure `/api/trpc` route is explicitly implemented and tested, and either implement `/api/auth/register` or remove unsupported signup flow; verify with browser-level system tests for login, onboarding, and one write mutation. Fifth, standardize test and CI pipeline: add root `test` scripts, add turbo `test` task, normalize package-level test scripts, and create GitHub workflows for lint, typecheck, test, and build; verify by running pipeline from clean clone and ensuring artifact logs are retained. Sixth, remove quality debt in core paths: disable `ignoreBuildErrors`, reduce `@ts-nocheck` in auth, middleware, routers, commands, and projectors first, and clean repository junk files; verify by strict build passing without newly introduced suppressions in those priority paths. Seventh, implement operations baseline: upgrade health endpoint to include DB and Redis checks plus projector status, add structured logging with error context, define initial alerts, and document backup and restore drill steps; verify by executing one simulated incident and documenting recovery time. Use this work order without reordering, because the sequence burns down exploit and data-loss risk first. At each checkpoint, if expected verification fails twice, escalate with a short issue packet containing failing command, file path, observed output, and attempted fix. Completion criteria are strict: all Critical and High findings closed, full checklist green, staging soak test stable for 24 hours, and documented evidence attached for each verification checkpoint.
